# Offline Analysis Project Direction

> 최종 업데이트: 2026-03-19
> 목적: Ritmo 앱과 별도로, 장기적으로 쓸 수 있는 오프라인 곡 분석 라이브러리 방향을 정리한다.
> 용도: 새 프로젝트를 열고 탐색/설계/프로토타이핑을 시작할 때 기준 문서로 사용

---

## 1. 왜 별도 프로젝트로 분리하나

현재 Ritmo 앱 안의 온디바이스 분석은 실험에는 성공했지만 서비스 품질 기준에는 못 미친다.

- 앱 릴리즈 일정과 분석 엔진 실험을 같은 저장소에서 묶으면 둘 다 느려진다.
- 분석 엔진은 앱 기능이 아니라 연구 개발 대상에 가깝다.
- 품질 향상을 위해서는 데이터셋, 벤치마크, 네이티브 구현, 평가 반복이 필요하다.
- 이 작업은 UI/플레이어 개발과 관심사가 다르다.

따라서 오프라인 분석 엔진은 별도 프로젝트로 분리하는 편이 맞다.

---

## 2. 문제 정의

글로벌 서비스 관점에서 중요한 상황은 다음과 같다.

- 모바일 업로드 품질이 나쁜 지역에서는 서버 분석 대기시간이 매우 길다.
- 오프라인 상태에서는 서버 분석 자체가 불가능하다.
- 라틴 댄스 음악은 범용 beat tracker로는 품질이 흔들린다.

우리가 풀고 싶은 문제는:

> "모바일 기기에서, 네트워크 없이도, 라틴 댄스 곡의 BPM/beat/downbeat를 실사용 가능한 수준으로 추정하는 엔진"

---

## 3. 프로젝트 목표

### 1차 목표

- 로컬 파일 입력만으로 BPM, beat timestamps, downbeat timestamps 반환
- 3분 내외 곡을 모바일에서 실용 시간 안에 처리
- 메모리 사용량이 앱을 죽이지 않을 수준으로 통제
- 서버 엔진과 비교 가능한 평가 체계 구축

### 2차 목표

- bachata/salsa에 특화된 보정 로직 추가
- 첫 박 기준 안정화
- phrase 추정에 도움이 되는 구조적 힌트 제공

### 3차 목표

- section/phrase/choreo assist까지 확장 가능한 코어 설계

---

## 4. 비목표

초기에는 아래를 목표로 잡지 않는다.

- 모든 장르를 잘 처리하는 범용 음악 분석 엔진
- 완전한 실시간 스트리밍 분석
- 앱 안에서 바로 production replacement
- section segmentation과 phrase boundary까지 한 번에 해결

초기 범위를 좁혀야 프로젝트가 살아남는다.

---

## 5. 추천 범위

### Phase 1 범위

- 입력: audio file or PCM
- 출력: `bpm`, `beats[]`, `downbeats[]`, `confidence`
- 장르 초점: bachata, salsa

### Phase 2 범위

- tempo ambiguity 보정
- bar alignment 보정
- genre-specific heuristics

### Phase 3 범위

- section hints
- phrase candidates
- user correction feedback loop

---

## 6. 권장 아키텍처

### A. 코어와 바인딩을 분리한다

권장 구조:

```text
offline-analysis-core
  - DSP / feature extraction
  - beat + downbeat engine
  - evaluation tools

bindings
  - iOS bridge
  - Android bridge
  - optional CLI / desktop benchmark runner
```

이렇게 분리하면 앱과 무관하게 엔진만 독립적으로 개선할 수 있다.

### B. 언어 선택

JS 단독보다는 네이티브 코어가 유리할 가능성이 높다.

후보:

- C++: DSP/모바일 배포 경험이 풍부함
- Rust: 메모리 안전성과 멀티플랫폼 관리에 유리

초기 판단으로는 Rust 또는 C++ 둘 다 가능하지만, 목표가 "모바일 배포 가능한 분석 코어"라면 JS 전용보다 네이티브 코어가 더 현실적이다.

### C. 입력 표준화

코어 입력은 가능하면 다음으로 통일한다.

- mono
- 22050Hz
- float32 PCM

이 전처리 규격은 서버와도 최대한 맞추는 것이 좋다.

---

## 7. 기술 전략 후보

### 전략 1. 규칙 기반 + DSP 보강

구성:

- onset strength
- tempo candidates
- beat tracking
- bar/downbeat heuristic
- bachata/salsa 특화 보정

장점:

- 초기 개발이 빠르다
- 데이터셋이 적어도 시작 가능하다
- 왜 틀렸는지 설명하기 쉽다

단점:

- 복잡한 리듬에서 한계가 빨리 온다

### 전략 2. 하이브리드

구성:

- DSP 기반 beat candidate
- 경량 ML로 tempo/downbeat re-ranking

장점:

- 완전 규칙 기반보다 품질 상한이 높다
- 모바일에서도 경량 모델은 가능하다

단점:

- 데이터셋 구축이 필요하다

### 전략 3. 엔드투엔드 ML

구성:

- waveform 또는 spectrogram 입력
- beat/downbeat 직접 추론

장점:

- 장기적으로 가장 높은 성능 가능성

단점:

- 데이터셋, 학습 파이프라인, 모델 경량화까지 난이도가 급상승
- 지금 바로 시작하기엔 리스크가 크다

현재 기준 추천은 전략 1로 출발해서 전략 2로 가는 방식이다.

---

## 8. 가장 중요한 것: 평가 체계

새 프로젝트의 핵심 산출물은 코드보다 먼저 벤치마크여야 한다.

최소 필요 요소:

- 라틴 댄스 곡 데이터셋
- 수작업 또는 신뢰 가능한 beat/downbeat 정답
- 서버 엔진 결과와의 비교 기준

핵심 지표:

- BPM error
- beat F-measure
- downbeat accuracy
- first-bar alignment stability
- 처리 시간
- peak memory

평가 없이는 "좋아진 것 같은 느낌"만 남고 프로젝트가 흔들린다.

---

## 9. 데이터 전략

초기 데이터는 아래 세 축으로 쌓는 게 좋다.

1. 현재 서버 분석 결과
2. 사용자의 수동 보정 결과
3. 별도 정답 레이블이 붙은 샘플셋

특히 사용자 보정 데이터는 장기적으로 가장 가치가 크다.

- 첫 박 수정
- BPM 수정
- phrase 재배치

이런 신호는 미래의 오프라인 엔진 보정 학습에 직접 연결될 수 있다.

---

## 10. 프로젝트 시작안

새 프로젝트를 판다면 첫 버전은 작아야 한다.

### 저장소 1차 목표

- CLI로 파일 하나 분석 가능
- 표준 입력 PCM 또는 WAV 지원
- bpm/beats/downbeats/confidence 출력
- 벤치마크 스크립트 포함

### 1차 마일스톤

1. 입력 표준화 파이프라인
2. baseline beat tracker
3. baseline downbeat heuristic
4. 평가 스크립트
5. 서버 결과와 비교 리포트

### 2차 마일스톤

1. bachata 보정 규칙
2. salsa 보정 규칙
3. 템포 반/배속 ambiguity 해결
4. 모바일 메모리 프로파일링

### 3차 마일스톤

1. 경량 ML re-ranker
2. 모바일 바인딩
3. Ritmo 앱 실험 통합

## 11. Ritmo와의 관계

이 프로젝트는 당장 Ritmo 앱의 분석 경로를 교체하는 용도가 아니다.

현재 권장 관계는 다음과 같다.

- 본 서비스: 계속 서버 분석 사용
- 새 프로젝트: 오프라인 엔진 연구용
- Ritmo 앱: 실험 브랜치에서만 통합 테스트

즉, 엔진이 충분히 검증되기 전에는 앱의 기본 경로를 바꾸지 않는다.

---

## 12. 리스크

- 데이터셋 없이 규칙만 만지다 품질 한계에 막힐 수 있다
- 너무 큰 범위를 잡아 끝까지 못 갈 수 있다
- 앱 통합을 너무 빨리 시도하면 연구가 기능 개발에 끌려간다
- "범용 음악 분석"까지 욕심내면 라틴 특화 강점을 잃는다

---

## 13. 추천 결론

새 프로젝트를 여는 건 좋은 선택이다.

다만 목표는 이렇게 좁혀야 한다.

> "Ritmo 앱과 독립된, 라틴 댄스용 오프라인 BPM/beat/downbeat 엔진을 만드는 연구 프로젝트"

이 정도로 한정하면 현실적이고, 성과도 측정 가능하고, 나중에 앱에 다시 붙이기도 쉽다.

---

## 14. 시작할 때 바로 정해야 할 것

- 코어 언어를 무엇으로 할지
- 입력 표준 포맷을 무엇으로 고정할지
- 평가용 곡셋을 어디서 만들지
- 성공 기준을 어느 수치로 볼지

이 네 가지가 정해지지 않으면 프로젝트가 금방 추상적인 아이디어 단계로 돌아간다.
