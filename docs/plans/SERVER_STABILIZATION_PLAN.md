# Ritmo Server Stabilization Plan

> 최종 업데이트: 2026-03-19
> 목적: 서버 기반 곡 분석을 유지하되 CPU/메모리 사용량을 줄이고 장애 가능성을 낮춘다.
> 대상: `server/` 작업을 맡을 Claude 또는 서버 최적화 담당자

---

## 1. 배경

현재 결론은 다음과 같다.

- 온디바이스 분석은 프로토타입 수준에서는 동작하지만 품질이 서버 분석보다 낮다.
- 당분간 서비스 품질 기준은 서버 분석이 책임진다.
- 실제 운영 리스크는 정확도보다도 서버의 CPU 사용량과 메모리 급증이다.
- 특히 오디오 파일 분석 중 전체 디코드, 중간 배열 중복, 파이프라인 동시 실행 때문에 메모리 피크가 크다.

즉, 지금의 핵심 과제는 "분석 품질 개선"보다 "같은 품질을 더 안정적으로 제공하는 구조"다.

---

## 2. 목표

### 1차 목표

- 분석 요청이 몰려도 서버가 죽지 않게 만들기
- 단일 분석 작업의 메모리 피크를 낮추기
- 같은 곡 재분석 비용을 줄이기
- 타임아웃, OOM, 장시간 점유 같은 운영 리스크를 통제 가능하게 만들기

### 2차 목표

- 평균 처리시간과 대기시간을 줄이기
- 서버 한 대당 처리 가능한 동시 작업 수를 예측 가능하게 만들기
- 추후 워커 분리나 수평 확장이 쉬운 구조로 바꾸기

---

## 3. 현재 병목 가설

실제 코드 측정 전이지만, 현재 구조에서 가장 가능성이 높은 병목은 아래와 같다.

1. 입력 파일을 원본 포맷 그대로 받아 분석 단계마다 다시 디코드한다.
2. `librosa`, `madmom`, waveform 계산, section 계산이 각각 큰 배열을 따로 잡는다.
3. 긴 곡을 한 번에 메모리에 올리고 처리한다.
4. FastAPI 요청 처리 컨텍스트 안에서 CPU 집약 작업이 직접 실행된다.
5. 워커 동시성 제한이 약하면 여러 요청이 한꺼번에 메모리를 잡아먹는다.
6. 동일 파일 재분석 시 캐시 hit 없이 같은 계산을 반복한다.

---

## 4. 기본 원칙

### 원칙 A. 업로드 직후 입력을 표준화한다

- 원본 MP3/M4A/FLAC를 바로 분석하지 않는다.
- 서버 진입 직후 `ffmpeg`로 분석 전용 표준 포맷으로 변환한다.
- 권장 표준 포맷: mono, 22050Hz, PCM WAV 또는 float32 WAV

효과:

- 후속 단계 메모리 사용량 예측이 쉬워진다.
- 분석 파이프라인이 포맷별 예외 처리에서 해방된다.
- waveform, beat, section 계산이 동일 입력을 공유할 수 있다.

### 원칙 B. 분석은 웹 요청에서 분리한다

- API 요청 스레드에서 분석을 직접 오래 돌리지 않는다.
- 업로드와 작업 실행을 분리하고, 분석은 별도 워커 프로세스가 수행한다.
- 최소한 프로세스 분리만 해도 OOM 이후 재시작과 격리가 쉬워진다.

### 원칙 C. 메모리 피크를 먼저 줄이고 그 다음 속도를 본다

- 현재는 CPU 최적화보다 메모리 안정성이 우선이다.
- 느리더라도 피크 메모리가 낮은 설계가 먼저다.

### 원칙 D. 한 번 계산한 결과는 최대한 재사용한다

- 파일 hash 기반 캐시를 강화한다.
- 원본 곡과 분석 전처리 산출물 둘 다 캐시 대상이 될 수 있다.

## 5. 권장 실행 순서

## Phase 1. 관측 가능성 확보

이 단계 없이 최적화에 들어가면 개선 여부를 판단할 수 없다.

- 분석 요청마다 `request_id`, `track_hash`, `duration`, `file_size` 기록
- 각 분석 단계별 소요시간 기록
- 각 단계 시작/종료 시 RSS 메모리 기록
- 타임아웃, 예외, OOM 직전 로그를 파일로 남기기

최소 로그 포인트:

- upload received
- ffmpeg normalize start/end
- beat analysis start/end
- waveform extraction start/end
- section analysis start/end
- response saved

최소 지표:

- 평균 처리시간
- P95 처리시간
- 최대 RSS
- 파일 길이 대비 메모리 사용량
- 캐시 hit ratio

## Phase 2. 입력 표준화

- 업로드 직후 `ffmpeg`로 mono 22050Hz WAV 생성
- 이후 모든 분석 로직은 이 표준 파일만 읽는다
- 원본 파일은 즉시 보관 또는 삭제 정책을 정한다

권장 효과:

- 디코드 로직 중복 제거
- 분석 모듈 간 입력 일관성 확보
- 포맷별 edge case 감소

## Phase 3. 워커 격리 + 동시성 제한

- FastAPI와 분석 작업을 분리
- 분석 워커 프로세스를 별도로 실행
- 서버 사양이 작다면 초기에는 동시 분석 수를 1 또는 2로 제한
- 대기열이 생겨도 서버가 죽지 않는 쪽이 우선

권장 구조:

```text
Client -> API -> Job Queue/DB status -> Worker Process -> Result Store
```

초기에는 큰 큐 시스템 없이도 가능하다.

- DB job row
- in-process scheduler 또는 별도 worker script
- job status polling

## Phase 4. 메모리 피크 절감

우선순위가 높은 항목:

1. `float64` 대신 가능한 곳은 `float32` 사용
2. 긴 신호 전체 복사 대신 slice/view 사용
3. 중간 결과를 재사용하고 불필요한 배열 즉시 해제
4. waveform 계산은 저해상도 peak만 남기고 원신호는 버리기
5. 필요 없는 단계는 옵션화해서 비활성화 가능하게 만들기

특히 확인할 것:

- `librosa.load()`가 기본 dtype과 resample에서 큰 메모리를 쓰는지
- madmom 입력 전후에 배열이 몇 번 복사되는지
- section 분석이 전체 신호를 또 한 번 잡는지

## Phase 5. 캐시 강화

- 파일 hash 기준 결과 캐시
- 표준화 WAV 캐시
- waveformPeaks 캐시
- section 결과 캐시

캐시 키 후보:

- file hash
- normalized audio hash
- analysis engine version
- parameter set

이렇게 해야 알고리즘 버전이 바뀌어도 캐시 오염을 막을 수 있다.

## 6. 구현 우선순위 제안

Claude에게 넘길 때는 아래 순서가 가장 현실적이다.

1. 분석 단계별 메모리/시간 로깅 추가
2. 업로드 후 `ffmpeg` 표준화 단계 추가
3. 분석 워커 프로세스 분리
4. 동시성 제한 및 작업 큐 도입
5. waveform/section 계산의 배열 복사 최소화
6. 캐시 구조 정리

이 순서의 장점은, 매 단계가 독립적으로 효과를 검증할 수 있다는 점이다.

---

## 7. 세부 제안

### A. 전처리 표준 파일

권장 명령 예시:

```bash
ffmpeg -i input.ext -ac 1 -ar 22050 -c:a pcm_s16le normalized.wav
```

주의:

- 너무 긴 파일은 표준화 단계에서도 디스크 용량을 먹는다.
- temp 파일 정리 정책이 필요하다.

### B. 메모리 안전한 실행 방식

- 분석은 요청 프로세스 안이 아니라 별도 프로세스에서 수행
- 프로세스 단위 timeout
- 작업 완료 후 프로세스 종료 또는 재사용 횟수 제한

실무적으로는 "worker가 몇 작업마다 재시작"만 해도 메모리 누적 리스크를 줄일 수 있다.

### C. 긴 곡 대응

- 파일 길이 상한선 설정 검토
- 우선 beat/downbeat를 먼저 계산하고 section은 후행 계산 가능하게 분리
- 프리뷰 품질과 전체 정밀 분석을 분리하는 것도 가능

### D. 서버 사양별 운영 모드

- 저사양 서버: 동시 분석 1
- 중간 사양 서버: 동시 분석 2
- 고사양 서버: beat/downbeat와 section을 별도 워커로 분리

---

## 8. 하지 말아야 할 것

- 정확한 측정 없이 알고리즘을 한꺼번에 갈아엎기
- API 요청 컨텍스트에서 무제한 분석 실행
- 표준화 없는 원본 포맷 직접 분석 유지
- 캐시 정책 없이 "일단 더 큰 서버"로 버티기

## 10. 성공 기준

다음이 만족되면 안정화 1차 성공으로 본다.

- 단일 분석 중 서버가 메모리 급증으로 죽지 않는다
- 3분 내외 곡 기준 처리시간 편차가 줄어든다
- 같은 파일 재분석 시 캐시 hit가 발생한다
- 동시 요청이 들어와도 대기열로 흡수되고 장애로 번지지 않는다

---

## 11. Claude 전달용 한 줄 요약

`server/`는 품질보다 안정화가 우선이다. 업로드 직후 오디오를 표준화하고, 분석을 별도 워커 프로세스로 격리하고, 단계별 메모리/시간 로깅과 동시성 제한부터 넣어라.
